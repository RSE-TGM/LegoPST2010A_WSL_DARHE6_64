SUMMARY

The MMI tool developed for the ENEL LEGOCAD environment is used to configure, display and manage plant animated synoptics.

MMI is available in two versions that differs in the way data collected and in data types representation.

LEGOMMI/simulator

MMI associated with a LEGOCAD simulator which sends data

LEGOMMI/SCADA

MMI associated with a SCADA (PSCS 2000 S.d.I.) which sends data and variable flags

 

In both versions data exchange between LEGOMMI and simulator or SCADA is structured using a client-server architecture where MMI is the client.

We consider three step in LEGOMMI use : configuration, compilation and synoptic animation.

The user may use two different executable tools:

config to built up the pages

mmi to use the animated pages during run-time

 

LEGOMMI config tool can also be used to develop pages which represent regulation schemes in order to obtain a regulation LEGO module (block) and to connect pages (i.e. schemes) among each other so as to create a regulation task. In this modality LEGOMMI can manage specific graphical regulation objects that provide connection capabilities and memorization of topological information. The same regulation scheme can be used in run-time mode as a plant synoptic for debugging purposes or plant monitoring.

LEGOMMI: versions available

MMI is available in two different versions depending on the data I/O software interface provided: the server from/to which LEGOMMI exchange data may be ENEL LEGO SIMULATOR or SCADA (actually S.d.I. SCADA PCS2000).

While in LEGOMMI/SCADA a specific composite object for alarm acknowledge and display is available in LEGOMMI/simulator alarm management is not provided.

The simulator sends all data as floating point variables, SCADA sends configured points data as floating point with some associated flags (for analog variables) or as status value with associated flags (for digital variables). In the LEGOMMI/simulator version, MMI flags manage additional information (such as limits thresholds surpass).

Information about connection simulator or SCADA variables is available in:

Flags consist of information, either on measure acceptability: out of scansion, out of reliability, forced; or warning flags, such as those concerning overcoming threshold or alarm acknowledge flag (RI).

Once an object has been linked with a simulator or SCADA variable, it receives the related data and the flags which causes modifications in objects colours and their aspect (i.e. color modification, displayed value modification etc.).

Object dynamic behaviour, due to linked variables values modification, depends on object type and user configured resources values. Colour change policy generally is the same for active objects: every flag set (on) is associated to a color defined by the user. The flags, RI and FZ, cause different behaviour on the object; RI causes color blinking between normal color and flag-on colour, FZ causes reverse representation realized using background color and flag-on colors.

A priority scale between flags effects is fixed; in fact, flags have not the same importance (for example "out of scansion" forces the related color regard less of the others). The priority scale of flags is the following:

LEGOMMI: page & libraries

A plant representation is composed by a series of graphical pages representing data and other schematic information during run-time (animation). Every page must be configured by instantiating some active objects, (icons, numeric display trends, bar chart ecc.) and by defining the background draw (not animated). During instantiation the user establishes the links between graphical behaviour of the active objects in animation and the values assumed by the simulator or SCADA variables in the run-time phase.

During page definition, the user has to assign the page type and geometrical page's characteristic. Three types of pages are available:

Synoptic Page:

used for the presentation of a plant synoptic; is completed by a series of pre-defined buttons on the left side of the window. The functions available through the push buttons are

- NOP

enables only the specific call-back of every object (i.e. buttons).

- INFO

info display on objects

- XAING

perturbation send (for simulator only)

- XSTAZ

rising of a station page associated with an icon.

- PLOT

plot (for trend display of the variable)

- CLOSE

closes the page

Synoptic Page:

same as synoptic but pre-defined buttons are not present.

Regulation Page

used for the construction of pages describing regulation schemes; in this case instantiation of objects, oriented to topological information memorization such as iconreg is allowed. During run-time is the same as synoptic.

LEGOMMI supplies a series of basic graphical objects which can be retrieved and instantiated by the user; these objects are organized in the standard active objects library named std.lib.

An active objects library contains and organizes a series of active objects, retrieved and reused by the user. It acts as a user-defined "toolbox" containing custom graphical objects. Customization of objects includes also the creation of "composite objects". Furthermore the user will be able to create a private library featuring pre-configured objects for speedy use. For example, a library can be dedicated to a particular kind of synoptic depending on the part of the plant under consideration (i.e. superheater, evaporator etc.).

Libraries, similarly to pages, are created by the user by instantiating objects and assigning the related properties (resources).

Every page is completely defined by these files:

<page>.pag

contains property for every instantiated object

<page>.bkg

contains property for the background draw

Every active objects library is defined by the file

<library>.lib

 

All file produced are in ASCII format and so cross-platform portability is guaranteed.

 

LEGOMMI: Context file

In order to start-up the configuration and the simulation activities both config and mmi need a configuration file (called Contex.ctx by default or <name>.ctx if precisely).

This file contains a number of basic information such as the list of pages and libraries that constitute the project; the directories where specific project file resides etc.

Some context information (i.e. directories names) can be changed when the user starts config by choosing "Modify Context" on the Context pulldown menu inside the main window. When the user starts up animation through the mmi, the ".ctx" file is loaded and the list of available pages is shown.

List of context fields:

Project

Directory containing the Context file.

Description

Application description.

Pages

Directory containing the pages.

Object library

Directory containing the active objects libraries.

Active icon library

Directory containing the animated icon's file.

Simulator

Directory containing the variable.rtf file. This file contains the list of all available variables in run time database.

Display list

List of LEGOMMI/simulator X-Server's display used in run time.

Scada Host name

Host from which Scada Will send data during the run-time phase. A number (the MMI progressive number connected with Scada) must follow the hostname.

Simulator Host name

Host from which the simulator will send data during run-time, local if it is not set.

When using mmi an added binary context file, named Context.ctx_rtf, is produced. This file contains the background information of all pages and in particular gathers the contents of all the background ASCII files in binary format to have these information stored in the memory and obtains a quick page-loading without loading in the page.bkg files. During start up, mmi controls if the Context.ctx and all the page.bkg files are older than the Context.ctx_rtf and loads the latter or creates a new one depending on test results.
