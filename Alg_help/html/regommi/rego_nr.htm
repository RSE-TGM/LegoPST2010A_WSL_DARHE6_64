Note 1

In a comparison between this module and the MOLT multiplier module (see the related documentation) it seems that the two modules execute the same operation: both provide as output the product of the two input. The basic difference consists in the names assigned to the variables. In fact, in the MOLT module the second input is called I2 whereas in the PROP module the second input (gain) is called KP. On the contrary this distinction, which does not affect a regulation scheme building with "LEGOCAD" method, has influence in building a regulation scheme with method "LEGOGRAF". The latter in fact, distinguishes, as follows, input and output in accordance with the name initial letter:

I: analogue input

U: analogue output

J: logic input/output

K: characteristic constant of the module (input for the source program)

To the analogue input identified by a letter "I", is associable an input gate which can be linked to the output of another module: therefore, such input, can be changed as user likes. On the contrary, the analogue input identified by a letter "K", can not be linked to an other module. In fact, they are considered as constants and to them is not allowed the linkage of an input gate; they can be modified only form outside the regulation scheme.

RAMP regulation module

 

The regulation module RAMP allows to generate a desired slope (KP/KT or -KM/KT), which moves the UA output from the tracking value II to a final value IN fixed in advance.

A distinction is now useful between the cases in which a stationary or transient condition is being calculating.

1) Stationary condition: if the module is in tracking phase (signal J1 true), the UA output is set to the tracking input value II, otherwise (J1 false) the output UA is set to the final value IN fixed in advance. In any case the two logical signals J3 (START/STOP logic in input) and JA (START/STOP logic in output) are set to the value "false".

2) Transient condition: if the module is in tracking phase (signal J1 true), the output is set to the tracking input value II, and the two logical signals J3 and JA are set to the value "false".

In opposite case (J1 false) the logical output UA copies the logical input J3; if JA proves false (the slope is not in START), the UA output keeps the value it had at the preceding time interval. Otherwise (JA true) the slope is in START and the UA output is turned into a slope, up to reach the desired final value IN.

About the output at the preceding time interval UAP a test is carried out; three cases are possible:

a) UAP < IN. The output is still less than the desired final value IN, and it is increased with the product deriving from the time interval Dt by the increasing gradient KP/KT:

UA = UAP + KP / KT * Dt

If the output UA exceeds the final value IN, it is forced to IN, and JA is set to the value "false" (the slope is in STOP).

b) UAP > IN. The output is still higher than the desired final value IN, and it is decreased by a product deriving from the time interval Dt by the decreasing gradient KM/KT:

UA = UAP - KM / KT * Dt

If the output UA is less than the final value IN, it is forced to IN, and JA is set to the value "false" (the slope is in STOP).

c) UAP = IN. The output at the preceding time interval is equal to the desired final value IN; it is consequently forced to IN, and JA is set to the value "false" (the slope is in STOP).

UA = IN

Finally we have some operations which are common to all possibilities previously defined:

1) Limitation of the UA output to become to the interval identified by the minimum and maximum values V.MIN and V.MAX (chosen by user).

2) Forcing of JA variable to the value "false" (slope in STOP), where the UA output exceeds the limits identified by V.MIN, V.MAX and its saturation is necessary.

RICO regulation module

 

The regulation module RICO sets the JU output to the "true" (or "false") value if the JI input maintains the "true" (or "false") value during at least a fixed time (I1 for "true" or I2 for "false"). More precisely, the logical output JU is set to "true" when the JI input, after a commutation from "false" to "true" (0. to 1., from which the name "rise front") maintains the "true" value for at least I1 seconds; the logical output JU is set to "false" when the JI input, after a commutation from "true" to "false" (1. to 0., from which the name "descend front") maintains the "false" value for at least I2 seconds. The operating way of RICO module is better explained in the fig. 1 diagram, where the commutations of the JU output are shown, as function of a certain time trend of JI input.

figure 1

 

Hereinafter a detailed description of the functionality of the RICO module follows.

Note that variables S1 and S2 are used to compute the elapsed time during a state of "true" or "false" respectively of the variable JI; and logical variables J1 and J2 are used too to show that during the last I1 or I2 seconds there has been a rise or descent front of JI.

1) Steady state: the JU output is set to the value of the JI input.

2) Transient: it is tested if there have been rise or descent fronts during the last two time steps (the actual one and the preceding one), using the value of the input at the preceding time step JIP.

2.1) Rise front on JI (JI "true" and JIP "false"). J1 is set at "true", J2 at "false" and S2 is set to zero.

The counter S1 (related to JI = "true") is increased with a quantity equal to a time interval:

S1 = S1 + Dt

If the counter S1 exceeds the interval I1 fixed in advance, the output JU is set to the value "true", J1 is set to "false" and the counter S1 is set to zero, otherwise no action can be engaged in.

2.2) Descent front on JI (JI "false" and JIP "true"). J1 is set at "false" J2 at "true" and S1 is set to zero.

The counter S2 (related to J1 = "false") is increased with a quantity equal to the time interval:

S2 = S2 + Dt

If the counter S2 exceeds the interval I2 fixed in advance, the output JU is set to the value "false", J2 is set to "false" and the counter S2 is set to zero, otherwise no action can be engaged in.

2.3) No rise or descent front on J1. The module looks at the signals J1 and J2 which can not be contemporary true.

2.3.1) Signal J1 true. The counter S1 (related to JI = "true") is increased with a quantity equal to the time interval:

S1 = S1 + Dt

If the counter S1 exceeds the interval I1 fixed in advance, the output JU is set to the value "true", J1 is set to "false" and the counter S1 is set to zero, otherwise no action can be engaged in.

2.3.2) Signal J2 true. The counter S2 (related to JI = "false") is increased with a quantity equal to the time interval:

S2 = S2 + Dt

If the counter S2 exceeds the interval I2 fixed in advance, the output JU is set to the value "false", J2 is set to "false" and the counter S2 is set to zero, otherwise no action can be engaged in.

2.3.3) Signals J1 and J2 false. No action can be engaged in.

RPIA regulation module

It is useful to define the different possibilities that RPIA module has depending on the value assumed by some of its logical variables:

1) RPIA in manual operating condition: J3 false.

1.1) J3 external balancing logic: true.

1.1.1) Stationary condition: the UA output is forced to the balancing or TARGET II value as well as the PI UP output:

UA = II

UP = II

1.1.2) Transient condition: the ER error, given from the difference between the II balancing value and the actuator output evaluated at the preceding time interval UAP, is calculated:

ER = II - UAP

If ER, as absolute value, is less than a maximum error fixed by user ERRMAX, the output is calculated by means of the following formula:

UA = UAP + ER / ERRMAX * Dt / TCORSA

where Dt and TCORSA respectively represent the time interval and the time (fixed by user) which actuator takes to cover its whole stroke.

On the contrary if ER, as absolute value, is equal or wider than ERRMAX, for output calculation the following formula is used:

UA = UAP + sign(ER) * Dt / TCORSA

in which the maximum gradient of output variation consists of the ratio Dt/TCORSA.

Finally, the PI output is set equal to the actuator output:

UP = UA

1.2) J1 external balancing logic: false.

1.2.1) Stationary condition: no calculation is carried out on the UA output which therefore keeps the value fixed by user during the filling phase of the file F14.DAT. The PI regulator output is set to the tentative value of UA output:

UP = UA

1.2.2) Transient condition: J4 and J5 logical signals (requests of output manual increasing and decreasing respectively) are compared; if they are equal (0 => no request, 1 => contemporary request of increasing and decreasing, both ignored) the preceding step value (UAP) which the UA output has is kept:

UA = UAP

If J4 and J5 are different, user tests which of them is "true":

a) J4 true (request of output increasing). The UA output is increased with the maximum allowable quantity in one time interval given by the ratio Dt/TCORSA (time interval and actuator stroke time respectively):

UA = UAP + Dt / TCORSA

the JE output signal (increase request satisfied) is set at the "true" value.

b) J5 true (request of output decreasing). The UA output is decreased with the maximum allowable quantity in one time interval given by the ratio Dt/TCORSA (time interval and actuator stroke time respectively):

UA = UAP - Dt / TCORSA

the JF output signal (decrease request satisfied) is set at the "true" value.

Finally, whatever combination the J3 and J4 logical signal have, the UP regulator output is set equal to the output:

UP = UA

2) RPIA in automatic condition: J3 true.

2.1) Stationary condition: no calculation is carried out on the UA output which therefore keeps the value fixed by user during the filling phase of the file F14.DAT. The PI regulator output is set at the tentative value of the UA output:

UP = UA

This expression for the UP output is deduced beginning from the expression indicated in first paragraph, deriving it as regards to the time and discretizing (it is useful to note that the IE and IZ input appear now normalized).

Subsequently, the ER error, given by the difference between the UP regulator output and the actuator output (UAP) evaluated at the preceding time interval, is calculated:

ER = UP - UAP

If ER, as absolute value, is less than a maximum error ERRMAX fixed by user, the output is calculated by means of the following formula:

UA = UAP + ER / ERRMAX * Dt / TCORSA

where Dt and TCORSA respectively represent the time interval and the time (fixed by user) which actuator takes to cover its whole stroke.

On the contrary if ER, as absolute value, is equal or wider than ERRMAX, for output calculation the following formula is used:

UA = UAP + sign(ER) * Dt / TCORSA

in which the maximum gradient of output variation consists of the ratio Dt/TCORSA.

Finally we have an operation which is common to all possibilities previously defined:

1) Limitation that UP regulator output has to belong to the time interval identified by the minimum and maximum saturation values SAT_MIN and SAT_MAX.

2) Limitation that UA actuator output has to belong to the interval (0, 1).

3) Calculation of JA, JB, JC and JD logical signals related to the full opening or closing of actuator stroke. Three cases are considered:

a) UA <= 0.02; the actuator is considered closed, therefore:

JA (logic of closed limit switch) true

JB (logic of not closed limit switch) false

JC (logic of opened limit switch) false

JD (logic of not opened limit switch) true

b) UA >= 0.98; the actuator is considered open, therefore:

JA (logic of closed limit switch) false

JB (logic of not closed limit switch) true

JC (logic of opened limit switch) true

JD (logic of not opened limit switch) false

c) 0.02 < UA 0.98: the actuator is considered neither closed nor opened, therefore:

JA (logic of closed limit switch) false

JB (logic of not closed limit switch) true

JC (logic of opened limit switch) false

JD (logic of not opened limit switch) true.

Variables initialization

 

During the calculation phase of a stationary condition, if balancing condition (J1 true) does not happen, output and input keep the tentative value assigned by user during the filling phase of the file F14.DAT.

Considering the function that the RPIA module has (an integral action on the IE input exists), user has to assign null value to the IE input in order to obtain that actual condition be stationary (see the theoretical treatment related to the PI industrial regulators).

Besides, since inside RPIA module the IE and IZ input are normalized dividing them for their end scale value (FSC_ER and FSC_IZ respectively), user has to assign to these two data a value different from zero in order to avoid numerical problems (division with null denominator).

For the same reason (numerical problems caused by division for zero), user has to assign not null values also to the TCORSA and ERRMAX data; in fact, they appears as denominator in some fractions (see above).
