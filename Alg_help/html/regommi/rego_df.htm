DIRA regulation module

 

The regulation module DIRA provides, as output, the input variable value (analogue) multiplied by a gain COSTAN fixed by user.

Once defined as II and UU, respectively the input and output analogue variable, in the formula we can write:

UU = COSTAN*II

Note 1

Inside the regulation schemes, the module DIRA is utilized for different purposes:

1) Analogue input block (no longer used: use instead analog interface)

2) Analogue output block (no longer used: use instead analog interface)

3) Trimmer controllable by station (see the documentation related to the carrying out station and to the regulation modules LOGO and LOGS)

4) Constant proportional gain (e.g. used to modify quantities measurement units)

DIVS regulation module

 

The regulation module DIVS provides as output the division result of the first input analogue variable by the second one.

Once defined as I1 and I2 the two input analogue variables and as UA the output analogue variable, in the formula we can write:

UA = I1 / I2

Note 1

In order to avoid that the output assumes too big values (for I2 = 0 UA tends to infinite), the module has been provided of a protection in case the second input (divisor) becomes very small. In this case, the input I2 is limited so that its absolute value can not become less than one thousandth.

DTRI regulation module

 

The regulation module DTRI provides, as output, the value of input variable (analogue) with a delay of one time interval.

Once defined as IN the input analogue variable, as UA the output analogue variable, as "t" the general instant time and as "Dt" the duration of time interval, in the formula we can write:

UA (t + Dt) = IN (t)

Note 1

In fact, if we look at the FORTRAN source of DTRI Module we note that the executed operation is only those to copy on the output UA the input IN:

UA = IN

From this point of view, the DTRI module is equal to the DIRA module with parameter COSTAN = 1 (see related documentation). The delay of one time interval, characteristic of the DTRI module, is obtained at SORTER level which recognizes such module (similarly to the DTRL) and deals with it in a different way as regards to all the other regulation modules.

Note 2

This module is typically used in feedback loops, where the signals return would cause some problems to the SORTER. In fact, it solves in a sequential way the equations constituting a regulation scheme (set of several regulation modules), so when a LOOP is present we have a conflict (the SORTER does not know what equation to solve first). To better clarify the problem we look at a simple example of regulation scheme consisting of two DIRA modules linked each other by the feedback:

figure 1

The equations constituting the scheme are (COSTAN = 1 for both the DIRA):

UA = IA

UB = IB

They might be solved if almost one equation would has the input known, but this does not happen for none of the two equations. In fact the input IA is equal to the output UB as the input IB is equal to the output UA.

In order to decouple the two equations, a delay in any point of the feedback loop is introduced (figure 2):

figure 2

The equations constituting the scheme are (COSTAN = 1 for both the DIRA):

UA (t + Dt) = IA (t + Dt)

UB (t + Dt) = IB (t + Dt)

UC (t + Dt) = IC (t)

in which is defined as "t" the general time instant, and as "Dt" the time interval duration.

In this case it is possible to solve the equations in a sequential way. In fact, it is enough to begin from the third, whose input IC(t) is equal to the output UB at the preceding interval, so it is known. We can easily understand that introduction of the DTRI module has carried out a temporal decoupling eliminating the conflict problems we had in case of figure 1.

FFDL regulation module

 

The regulation module FFDL simulates the behaviour of a bistable multivibrator of DELAY type (delay of one time interval). Precisely, the logical output JU at the time is equal to the input JI at the preceding time instant t - Dt (Dt = time interval):

JU(t) = JI(t - Dt)

In figure 1, as example, a possible run diagrams related both to the input JI and consequently to the output JU is shown:

figure 1

Operatively the time interval delay is obtained through the introduction of a JS state which acts as de-coupler between input and output. Examining more attentively the functioning of this module we have:

1) Stationary condition: the output JU copies the input JI.

2) Transient condition: the output JU copies the state JS (initialized to the value "false") and the state JS copies the input JI.

FFFS regulation module

 

The regulation module FFFS simulates the behaviour of a bistable multivibrator of SET-RESET type sensitive to the rise fronts. More precisely, when the SET input JS changes over from "false" to "true" (or from 0 to 1 ---> rise front) the logical output JQ is forced to the value "true", while it is forced to the value "false" when the RESET input JR changes over from "false" to "true" (or from 0 to 1). In all other cases JQ keeps the preceding value. Summing up:

JQ = true, if JS changes over: false ==> true

JQ = false, if JR changes over: false ==> true.

Besides the output JQ, also the output JN is present and this is always equal to JQ negation. From what previously said, it is clear that the output can change over the value only when on JS and/or JR input some "rise fronts" are present (change over from "false" to "true").

In figure 1, as example, a possible run diagram both related to JS, JR input and consequently of the JQ output are shown (JN is not shown).

figure 1

We can examine more in detail the operation carried out by this module.

1) Stationary condition: no operation is carried out, the output JQ keeps the tentative value assigned by user.

2) Transient condition: using some input to the preceding time intervals JSP and JRP, user has to consider if, in correspondence with the last 2 time instants (the preceding and the present), there are some "rise fronts". Four cases are possible:

a) No rise front on JS and JR: no operation is carried out.

b) Rise front only on JS (JS "true" and JSP "false"): JQ is forced to "true".

c) Rise front only on JR (JR "true" and JRP "false"): JQ is forced to "false".

d) Contemporary rise fronts on JS and JR: JQ is forced to "false" as for case c).

FFSR regulation module

 

The regulation module FFSR simulates the behaviour of a bistable multivibrator of SET-RESET type. More precisely, when the two input JS and JR are equals (both true or false), the logical output JQ keeps the value; it is forced to a value fixed in advance when only one input is true:

JQ = true, if JS = true (JR = false)

JQ = false, if JR = true (JS = false)

Besides the output JQ, also the output JN is present and this is always equal to JQ negation.
