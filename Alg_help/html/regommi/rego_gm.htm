GFUN regulation module

 

The regulation module GFUN is a function generator with NP sharp points, where NP is included within 2 and 10, which operates as follows: defined as I1 the input signal and as (X(1), Y(1), X(2), Y(2)), ..., (X(NP), Y(NP)) the NP couples of points on the X-Y plane (which will identify the NP sharp points) with X(1) < X(2) < ... X(NP), the output UA is:

UA = Y(1), for II < X(1)

UA = lin.funct. of II passing through (X(1), Y(1)), (X(2), Y(2)), for X(1) < II < X(2)

UA = lin.funct. of II passing through (X(2), Y(2)), (X(3), Y(3)), for X(2) < II < X(3)

...

UA = lin.funct. of II passing through (X(NP-1), YNP-1)), (X(NP), Y(NP)),
for X(NP-1) < II < X(NP)

UA = Y(NP), for II > X(NP)

It is evident that we can have different broken line diagrams depending on mutual ratio between Y(1), Y(2), ..., Y(NP) (see figure 1).

Physical and geometrical data

 

In addition to the initial values of input and output variables, whose meaning has been defined above, user must assign the following data set to complete the INPUT required by the GFUN module:

 

[] NPUNTI number of NP sharp points characterizing the function (it has to be included within 2 and 10)

[] Xi abscissa X(i) of the i-th sharp point (i = 1, ..., 10)

[] Yi ordinate Y(i) of the i-th sharp point (i = 1, ..., 10)

 

Independently of the number of sharp points NP, fixed by user, it is useful to note that within the set of data ten couples of points always appear.

Among these points, only the first NP are significant, the other being totally ignored by module. So, user can assign to the irrilevant couples of points any value (also no value).

 

Variables initialization

 

During the initialization phase of the GFUN module variables, user must not pay particular attention.

On the contrary, during the data assignment phase some precautions are needed:

1) number of sharp points NP included within 2 and 10

2) first NPUNTI abscisse X1, X2, ..., XNPUNTI such as to respect the condition:

X1 < X2 < ... < XNPUNTI

The non-respect of this condition can casue errors.

GPRO regulation module

 

The regulation module GPRO is a function generator with two sharp points which operates as follows:

defined as II the input signal and as (X0, Y0), (X1, Y1) the two couples of points on the plane X-Y (which will identify the two sharp points), where X0 < X1, the UA output is:

UA = Y0, for II < X0

UA = Y1, for II > X1

UA = lin.funct. of II passing through (X0, Y0), (X1, Y1), for X0 < II < X1

LOG0 regulation module

 

In all cases in which user has to use the STAZ program to simulate the behaviour of actual actuator stations (see related documentation), the regulation module LOG0 (or LOGS) must be utilized. In fact, this module consists of a logic block presented between an actuation station and a regulation in the strict sense of the word.

It is important to point out that STAZ program only operates within the "CAD SIMULATOR" environment (LEGOCAD environment allowing to simulate a plant functioning by means of a mathematical model consisting of sub-models, or tasks managed in a centralized way by SCHEDULER). Consequently, it makes sense to insert LOG0 module in a regulation scheme only whether user aims to simulate, in "CAD SIMULATOR" environment, a model consisting of (besides the regulation scheme), almost an actuation station (STAZ program).

Hereinafter, we examine the function that this module has, postponing in the following cards the description of its possible uses.

We begin pointing out the different possibilities that module has according to the values of some logical variables:

1) Stationary condition: the JS logical output (logic state toward the regulation) is set equal to the state requested by the JR station. Besides the JE output is forced to the "false" value (JE becomes "true" when a request coming from station is examined, on the contrary it is "false").

JS = JR

JE = false

2) Transient condition.

2.1) JE true: the request coming from station has already been examined; therefore JE signal is equated to the "false" value.

2.2) JE false: the request coming from station has not been examined jet.

2.2.1) JS = JR: the state JS is equal to the JR request; no operation is carried out.

2.2.2) JS < > JR: the state is different than the request; the state JS if forced so that it becomes equal to JR; besides, JE is set on "true" (request coming from examined station):

JS = JR

JE = true

Now we analyse the cases in which the module LOG0 is used. Since it acts as interface between station and regulation, it is better to examine the different types of station foreseen one by one, looking at the interconnecting schemes and related regulation blocks.

JS = JR

JE = false

2) Transient condition.

2.1) JE true: the request coming from station has already been examined.

2.1.1) JS = JR: the already examined request has been met; therefore JE signal is indicated to the "false" value.

2.1.2) JS < > JR: the request, already examined, has not been met; no action is carried out so that JE keeps al the "true" value.

2.2) JE false: the request coming from station has not been examined jet.

2.2.1) JA true: a forcing is enabled.

2.2.1.1) JS = JF: the state is equal to the forcing. JS and JR are compared: if they are different JE is forced to the value "true" (examined request), if they are equals no action is carried out (JE keeps its preceding value).

2.2.1.2) JS < > JF: the state is different than the forcing value. JS is forced to the JF value and JE is forced to the value "true" (examined request).

2.2.2) JA false: no forcing is enabled.

2.2.2.1) JS = JR: the state JS is equal to the JR request; no operation is carried out.

2.2.2.2) JS < > JR: the state is different than the request; the presence of interblocks is examined (variable JI):

a) JI true: no interblocks, the state JS if forced so that it becomes equal to JR; besides, JE is set on "true" (request coming from examined station).

JS = JR

JE = true

b) JI false: interblocks are present, JS keeps to its value (different from JR), nevertheless JE is set to the value "true" (request coming from examined station, but not met because of interblocks).

MEMA regulation module

 

The MEMA regulation module simulates an analogue memory with automatic/manual station.

In normal functioning, the analogue memory copies on the output UA the value of the automatic input IA or of the manual input II (target), according to the logic input J3 (A/M logic input).

Moreover, the output value may be forced to that of the input IB (if the logic input JB is "true"), regardless of the state of the memory (automatic or manual).

The value of the automatic input may be scaled by the multiplying factor K_INGA.

The operating strategy of the MEMA module is furthermore described.

1) Forcing imposed: JB true.

The output UA is set equal to the forcing input IB:

UA = IB

2) Normal operating conditions: JB false.

2.1) Automatic functioning: J3 true.

The output UA is set equal to the product of the automatic input IA by the coefficient K_INGA (assigned by the user):

UA = K_INGA * IA

2.2) Manual functioning: J3 false.

2.2.1) Logic "target-value" J1 true.

The output UA is set equal to the target value II:

UA = II

2.2.2) Logic "target-value" J1 false.

2.2.2.1) Steady state: no calculations are performed, and the output UA maintains its value, as imposed by the user in the file F14.DAT.

2.2.2.2) Transient: the logic inputs J4 (manual increase request button) and J5 (manual decrease request button) are compared; if they are equal (0 => no request, 1 => contemporary request of increase and decrease, both discarded) the output UA maintains the value at the preceding time step (UAP):

UA = UAP

If J4 and J5 are different the module tests which one is true:

a) J4 true (manual increase request button). The output UA is increased by the product of the time step Dt and the gradient KK:

UA = UAP + KK * Dt

the logic output JE (manual increase request satisfied) is set to true.

b) J5 true (manual decrease request button). The output UA is decreased by the product of the time step Dt and the gradient KK:

UA = UAP - KK * Dt

the logic output JF (manual decrease request satisfied) is set to true.

MEMB regulation module

 

The regulation module MEMB is a memory block functioning as follows:

1) When the switching logical signal AUTOMATIC/MANUAL J3 (automatic) is true, it copies on the UA output the automatic input IA

2) When the switching logical signal AUTOMATIC/MANUAL J3 (manual) is false, it keeps the output blocked.

Therefore we can conclude that the MEMB module behaves as memory (hold) only when J3 is false, and as tracker (sample) when J3 is true.

MIST regulation module

 

The regulation module MIST simulates the behaviour of a thermocouple for water and steam. It takes as input the pressure IP [Pa], the enthalpy IH [J/kg] and the flow rate IW [kg/s] of the fluid, and produces as output the estimated temperature UT [C].

This module contains great simplifications and should be considered temporary, until a new one, able to describe some phenomena here neglected, will be developed.

It is supposed that the estimated temperature UT is in relationship with the real one IT (obtainable by means of the steam tables, knowing enthalpy IH and pressure IP) by a transfer function of a low-pass filter (see the theoretical discussion in the FILT module):

where s is the Laplace operator, IT(s) and UT(s) are the Laplace transforms of the real temperature IT(t) and of the estimated one UT(t), TMIS is the time constant of the filter. The latest is not constant but it is considered inversely proportional to the fluid flow rate IW. Assuming that the relationship TMIS - IW does not depend on the direction of the flow, and limiting the time constant in order to avoid it reaches too large values when the flow rate comes near zero, one gets a graphic of the relation TMIS - IW as shown in figure 1.

 

figure 1

 

The following formulae make clear what previously said.

First of all we have the actual temperature calculation of fluid IT, using the steam tables:

IT = f (IP, IS)

where IS is the fluid entropy, also obtainable through the steam tables:

IS = f (IP, IH)

A distinction is now useful between the cases in which a stationary or transient condition is calculating:

1) Stationary condition: the UT output is set equal to the calculated actual temperature IT:

UT = IT

2) Transient condition: using the following formula the UT output is calculated:

UT = TMIS / (TMIS + Dt) * UTP + Dt / (TMIS + Dt) * IT

where Dt and UTP respectively represent the time interval and the output to the preceding time interval. This expression is obtained beginning from the transfer function previously mentioned, passing from the domain of Laplace transforms to the temporal domain and discretizing.

We consider now the time constant expression TMIS which is given by the product of a constant term T_MISU (fixed by user), for a term inversely proportional to the IW flow rate and limited on the upper part to the value 100:

TMIS = T_MISU * min (W_NOM./IW, 100)

where W_NOM. and T_MISU respectively represent the nominal values of flow rate (fixed by user), and of time constant (when the flow rate IW is equal to the nominal one W_NOM. the time constant is equal to T_MISU). This formula is clearly in accordance with the diagram shown in figure 1, where we set:

Tmax = T_MISU * 100

IWmin = W_NOM. * 1 / 100
