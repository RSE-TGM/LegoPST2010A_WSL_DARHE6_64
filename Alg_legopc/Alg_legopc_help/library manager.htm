Modules instancing

 

HOW TO INCLUDE A LEGO MODULE INTO A COMPONENT LIBRARY

 

 

Introduction: what is a module instance?

 

A newly created module requires, before it can be used inside LegoPC, a certain amount of operations. These operations take the name of instantiation and allow:

 

ü      associating the module with one or more icons, collected in graphical libraries,

 

ü      associating a certain number of physical ports to each icon,

 

ü      matching each physical port with the corresponding module variables.

 

Even already existing library modules may need extra instantiations. In facts, library modules may be extremely generical. For instance, a heat exchanger (i.e. module EXCH) can be divided into a number of longitudinal cells ranging between 2 and 6, and the wall of each cell can be in contact with one or more flue gas sections, each one having a different temperature. If this approach is very flexible in consideration of the often not simple topology of the heat exchangers in real power plants, the CAD-like approach used by LegoPC (based on a topological connection by physical ports) needs to define before using the module, how many variables it contains. So we need an EXCH instance with two cells, one with three, etc. The standard graphical libraries provided with LegoPC cope with the most common usages of the module. Suppose, however, you need using an EXCH with four cells and the second cell exchanging with two sections of the flue gas duct at different temperatures. In that case you’ll have to make an “ad hoc” instance of EXCH.

 

Fig. 1 – EXCH instances with three and four cells

 

The present notes show all the steps to be accomplished for instancing a LegoPC module.

Provided you want to make an instance of a module that is already present in the modules library (either a standard one or one you have added), four steps must be accomplished:

 

1.      Add icons of components to be instanced

 

2.      Make the so-called instantiation first phase, associating your instance with a fixed set of variables

 

3.      Define (if needed) new connectors to be subsequently associated to the component

 

4.     Make the so-called instantiation second phase, setting up physical ports and associating each of them with a set of variables.

 

All these phases are integrated in a unified interface, allowing to perform the right phase at the right time.

 

Fig. 2 – Librarian application: main window

 

The interface for creating modules instances can be called from the Module menu of the librarian application (fig. 2) after selecting the listbox item corresponding to the considered module. Supposing you want to create a new instance for the module EXCH, the window shown in fig 3 appears.

 

Fig. 3 – preinst main window

 

 

You can recognize the following elements inside the window:

·        A void Library entry that will contain the name of the instanced graphic objects library, your instance is going to be part of. Don’t insert the name by typing directly in the entry widget: you can use either the button Choose (to select an existing library) or the button New Library to create a new one. The latter is abilitated only for new instances. All the libraries are subdirectories of the directory pointed by the LG_LIBRARIES environment variable.

·        A default image, that will be associated automatically to the component if you don’t provide it with an icon.

·        A combo box Instance, allowing you to select a already existing instance. Whatever their objects library is, all existing instances appear listed by increasing module instance name. A “module instantiation name” is an alpha-numeric tag constituted by the four letters of the module name followed by a two-digit number identifying the current instantiation (e.g. exch_0, exch99, etc).

·        A comment entry giving a short identification of the characteristics of the current instance.

·        Four buttons at the bottom allowing to modify the current instance (Edit), to exit the instantiation (Cancel), to copy icons and instantiation information up to the first phase of the current instance on another graphical library (Copy To) and to create a new instance (New Instance). Both Copy to and New Instance generate automatically the new instantiation name by searching the first available. Beware not to mix the Copy to with New Instance operations because in some cases the generated instance name is not unique.

 

So, if you want to modify an existent instance, the first operation is to chose the instance to be modified with the combo box Instance. The result is shown in fig. 4. The fields Library and Comment are automatically filled up and the component icon is shown.

 

Fig. 4 – loading an existent instance

 

If you want, instead, to create a new instance, press the button New Instance. In this case only a new unique instance name will be provided. Thereafter, you must type in the comment and associate the new instance to an existent library or create a new one.

Both modifying old instances or creating new ones, at the end the button Edit becomes active. By selecting it, you obtain the Edit interface, shown in fig. 5.

 

Fig. 5 – the Edit interface

 

The Edit interface lists all the important phases of a module instancing. For each phase, the button Help gives a short explication, the central button activates it and the checkbutton on the left shows whether this phase has already been performed.

 

Let’s see the meanings of each phase.

 

1.      Icon editing

 

Each instance is associated with one or more icons. This can be made with many commercial programs (e.g. PaintShop PRO). When you press the button Icon Editing the program pointed to by the LG_ICONEDITOR environment variable is called. Whichever program you want to use there are some standards you must comply with:

 

·        Icons must be in gif-compuserve format (version 89a- non interlaced) and you must set the transparency flag. With PaintShop PRO this option is named: “set the transparency value to the background color” and means that when you fill an area with the background color, that area will be visualized to be transparent.

 

·        At least one icon must exist, named as the module instantiation name followed by the orientation attribute “n” (as “nord”), e.g. exch_0n.gif, exch99n.gif. This icon must be placed in the graphical library (subdirectory of LG_LIBRARIES), which your new instance must belong to.

 

·        To be able to exploit the possibility to rotate the drawing during topology construction, you must provide three other files with the same name but orientation attribute “w”,”s” and “e”: the contained image must be the original one (“n”) rotated for multiples of 90°.

 

·        If you don’t create at least a “nord” icon for the current instance, a default icon (a white filled rectangle) will be automatically associated at the end of the first instancing fase.

 

 

2.      New Connectors

 

Connectors (physical ports) are arrow symbols containing an array of slots associated each with a module variable. When you create a model with the LegoPc topology manager, you connect two modules by drawing a line binding two connectors. The module variables are automatically connected  slot by slot making use of their positional correspondence.

Connectors are normally associated one to each homogeneous group of variables: hydraulic variables, thermic variables, electrical variables and so on.

For new applications you can need to create new connectors. By pressing on the corresponding button, the interface shown in fig. 6 is visualized. You must type:

·        A Name specifying a (long) connector description that will appear as help on the connector itself by right-clicking on the connector symbol in the toolbox during the second instantiation phase.

·        A 4-chars-long Short Name identifying all the files associated to the connector

·        A Line color to be used by the LegoPc topology manager to draw lines connecting this connector type

·        Whether the line of the previous point must be dashed

·        How many slots the new connector must have.

 

Fig. 6 – New Connectors interface

 

Provided this information, you have to press on the Insert button. The interface in fig. 7 is shown, allowing to specify for each slot a string identifying the type of variable associated.

 

Fig. 7 – New Connectors slots definition

 

By pressing on Insert, the new connector is created.  Moreover, 12 graphical files associated to the connector are automatically placed in the LG_PIXMAPS location. In these files the connector is white-filled. To change the filling color, edit the files with a normal icons editor  (e.g. the same used for the instance gifs).

 

For details on the files created by connectors definition phase, see section “Adding new ports” of the LegoPc hyperhelp.

 

3.      First Phase

 

The first instantiation phase associates a set of variables to the module instance. By pressing on the corresponding button, an interface appears (fig. 8). Some questions to be answered are shown, the same posed by the I3-I4 routines of the module to be instanced.

 

Fig. 8 – Interface of the first instantiation phase

 

The button Send validates the answer typed in the entry, while Cancel quits the first instantiation phase.

The questions shown are:

 

ü      Some specific questions posed by the module (routine I4) in order to define a unique set of states, algebraic outputs and input variables. Among these: the number of cells and the number of exchanging surfaces for each cell.

 

ü      ALTRE CONFIGURAZIONI DA PREINSTANZIARE [S/N]: you can answer S (yes) or N  (no). You answer yes only if your module has multiple configurations. Modules with multiple configurations contain one algebraic relationship that can be used to calculate more than one algebraic output variable. For instance, the library module VALV has an algebric relationship binding mass flow rate, upstream pressure and downstream pressure. As there is no causality among the three variables, each one can be considered as the associated output variable. So, VALV (as some other library modules) allows you to chose what variable must be considered as output. During FORTRAN instantiation you can leave this choice undefined: the module topology constructor will decide the best configuration for you. In this case you can answer S (yes) and the module questions are repeated again and again till you answer N (no). Each time you’ll answer the questions with a different topological possibility (mass flow rate, upstream pressure or downstream pressure).

 

4.      Second Phase

 

 

Fig. 9 – The “Module inst” application

 

The last phase allows matching the “nord” orientation gif with physical ports and, for each physical port, to match the corresponding slots with LegoPC variables.

Selecting the “second-phase” button, a graphical application constituted by two windows will appear: the first shows the drawing of the instance “nord”, while the other (“toolbox”) contains the connectors palette.

You can select a connector type (port) inside the toolbox window. Standard coding for ports colors is:

yellow     =   water fluidodinamic port

gray        =   flue gas fluidodynamic port

green      =   thermic exchange port

cyan       =   command port

magenta  =   electrical port

orange     =   generical port (a slot with ten variables)

Chose right port and orientation by clicking on it in the “toolbox” window, then paste it upon the drawing with a mouse CTRL + Left click.

 

In case of wrong orientation select an arrow in the toolbox and then CTRL + Left click on the port to be rotated. To clear a not needed port select the “X” tool and CTRL + Left click on the port to be erased.

 

Ports can be dragged at will on the drawing.

 

To match a port with variables, double-click the port. A new window will appear with a list of slots for each slot to be matched with a variable. You need to select the slot and then search the corresponding variable inside the list-box on the window right.

 

To save all the information, select File/Save.

 

If the second phase for the current preinstantiation has been already performed, the modifications begin from the last saved situation.

 

Right-clicking on a connector in the toolbox window visualizes its long name as a help facility on the connector itself.
